# Теория MongoDB для бэкенда

## Что такое MongoDB
MongoDB — это документо-ориентированная NoSQL база данных. Данные хранятся в коллекциях в формате BSON (бинарный JSON), а не в таблицах и строках, как в реляционных СУБД.

## Когда MongoDB подходит
- Быстро меняющаяся структура данных.
- Большое количество полуструктурированных данных.
- Высокая скорость разработки API.
- Необходимость горизонтального масштабирования.

## Базовые сущности
- `database` — логический контейнер коллекций.
- `collection` — набор документов (аналог таблицы).
- `document` — JSON-подобный объект (аналог строки).
- `_id` — уникальный идентификатор документа.

## Моделирование данных
### 1) Embedding (вложение)
Связанные данные хранятся в одном документе.
- Плюс: меньше запросов, быстрые чтения.
- Минус: документ может сильно разрастаться.

### 2) Referencing (ссылки)
Связь хранится через идентификаторы других документов.
- Плюс: гибче при больших связанных сущностях.
- Минус: для сборки данных нужны дополнительные запросы или aggregation.

Для бэкенда обычно используется смешанный подход.

## CRUD-операции
- Создание: `insertOne`, `insertMany`.
- Чтение: `find`, `findOne`.
- Обновление: `updateOne`, `updateMany` с операторами (`$set`, `$inc`, `$push`).
- Удаление: `deleteOne`, `deleteMany`.

## Индексы
Индексы ускоряют чтение, но увеличивают стоимость записи.

Ключевые типы:
- Обычные (`{ field: 1 }`).
- Составные (`{ userId: 1, createdAt: -1 }`).
- Уникальные (`unique`) для email/username.
- TTL для автоматического удаления временных данных.
- Text для полнотекстового поиска.

## Aggregation Pipeline
Инструмент для сложных выборок и аналитики.

Частые этапы:
- `$match` — фильтрация.
- `$group` — группировка и агрегирование.
- `$project` — выбор/преобразование полей.
- `$sort` и `$limit`.
- `$lookup` — связь между коллекциями.

## Транзакции
MongoDB поддерживает ACID-транзакции (в replica set). Они нужны, когда изменения в нескольких документах должны быть атомарными.

## Надежность и масштабирование
- **Replica Set**: primary + secondary, автоматический failover.
- **Sharding**: распределение данных по шардам для масштабирования.

## Безопасность
- Включать аутентификацию и роли (RBAC).
- Ограничивать сетевой доступ к БД.
- Использовать TLS.
- Делать регулярные бэкапы и проверять восстановление.

## Практики для backend
- Проектировать схему от реальных запросов.
- Создавать индексы под самые частые фильтры и сортировки.
- Использовать UTC для дат.
- Настроить валидацию документов на уровне коллекции.
- Мониторить медленные запросы (`explain`, profiler).
