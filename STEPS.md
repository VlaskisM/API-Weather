# С чего начать реализацию Weather API

Краткий порядок работ: сначала «скелет» и окружение, затем данные и внешний API, потом бизнес-логика и HTTP. Подробности по слоям и моделям — в [PLAN.md](PLAN.md).

### Если привыкли к PostgreSQL + SQLAlchemy

Тут **ничего не противоречит** привычному `DATABASE_URL` в Pydantic Settings: по смыслу то же — одна строка (или сборка из полей) в конфиге, только переменные для MongoDB (`MONGODB_URL`, имя БД).

| Было (Postgres) | Стало (Mongo в этом проекте) |
|-----------------|------------------------------|
| `create_engine` / async engine + `sessionmaker` | `AsyncIOMotorClient` + `init_beanie` |
| SQLAlchemy `Base`, таблицы, Alembic | Beanie `Document`, коллекции; индексы в коде Beanie или скриптами |
| Сессия на запрос / dependency | Запросы через Beanie (или Motor) без «сессии» в SQL-смысле |

**Про `lifespan`:** с **async** Postgres на SQLAlchemy 2 тоже часто делают ровно это: при старте приложения создать engine и пул, при остановке — `dispose()`. Другое дело, если вы раньше работали со **sync** SQLAlchemy и `create_all()` при импорте — тогда отдельный lifecycle не напрашивался. Для Motor/Beanie явное «открыл при старте uvicorn, закрыл при shutdown» — нормальная и предсказуемая схема, не замена Settings, а **когда** применить URL из Settings.

## Перед кодом (один раз)

1. **Python** — версия 3.11+ (удобно для async и типов).
2. **MongoDB** — локально (Docker или установка) или [MongoDB Atlas](https://www.mongodb.com/cloud/atlas) — чтобы сразу проверять подключение.
3. **Ключ OpenWeatherMap** — зарегистрироваться на [openweathermap.org](https://openweathermap.org/api), взять API key (для геокодинга и текущей погоды).
4. **Виртуальное окружение** — `python -m venv .venv`, активировать, дальше все зависимости только в него.

## Шаг 1. Репозиторий и зависимости

- Создать `requirements.txt` по разделу из [PLAN.md](PLAN.md) (fastapi, uvicorn, motor, beanie, httpx, pydantic-settings, python-dotenv, pytest, pytest-asyncio).

**Готово, когда:** зависимости ставятся без ошибок, `.env` читается (пока можно не использовать в коде).

## Шаг 2. Минимальный FastAPI и конфиг

- Пакет `app/`, `main.py` с `FastAPI()`, корневой маршрут вроде `GET /health` → `{"status": "ok"}`.
- `config.py` — `pydantic-settings`: общие поля приложения и, по желанию, отдельная модель для Mongo (например `MongoSettings` / блок в `Settings`) с полями вроде `mongodb_url`, `mongodb_db_name` — строка подключения и имя БД задаются **только здесь** (из `.env`), при необходимости URL можно собрать из частей (хост, порт, user/password) в `model_validator` или через computed field. Остальное: OWM ключ, TTL, интервал фона.

**Готово, когда:** `uvicorn app.main:app --reload` поднимается, `/health` отвечает.

## Шаг 3. Подключение MongoDB (database + models)

**Где URL.** Строка подключения и имя базы — в **Pydantic Settings** (шаг 2). **Docker** лишь поднимает `mongod`; хост в URL прописывается в `.env`, который читает тот же Settings.

**Нужен ли именно `lifespan` — нет.** Нужен **какой-то** момент, когда вызвать `AsyncIOMotorClient` и `init_beanie`; `lifespan` — один из вариантов (удобно держать startup/shutdown в одном месте). Альтернативы:

| Вариант | Суть |
|--------|------|
| **`lifespan`** | Один контекст-менеджер: yield между init и shutdown, закрытие клиента гарантировано при остановке uvicorn. |
| **`@app.on_event("startup")` / `"shutdown"`** | То же по смыслу, только два обработчика. В новых проектах FastAPI чаще рекомендуют `lifespan`, но события всё ещё работают. |
| **Инициализация при импорте** | В конце `database.py` или в `main.py` после `get_settings()` один раз вызвать `init_db()` — проще для прототипа. Минус: при `--reload` возможны лишние подключения; корректный `client.close()` на выходе тогда через `atexit` или всё же `shutdown`-хук — иначе соединения могут висеть до завершения процесса. |

Итого: **без `lifespan` можно**, если вы явно где-то делаете `init_beanie` и по возможности `close()` при остановке приложения.

- `database.py` — функции вроде `init_db(settings)` / `close_db`, которые принимают настройки и создают `AsyncIOMotorClient` с URL из Settings; вызывать их из выбранного варианта выше (`lifespan`, `on_event` или при старте модуля).
- `models.py` — два Beanie-документа: `City`, `WeatherRecord` (поля как в [PLAN.md](PLAN.md)); для `name` у города — уникальный индекс во вложенном классе **`Settings` у документа Beanie** (не путать с Pydantic `Settings` приложения).

**Готово, когда:** при старте приложения нет ошибок подключения; в MongoDB появляются коллекции после первой записи (или после тестового `insert` в консоли/скрипте).

## Шаг 6. Unit of Work и зависимости FastAPI

**Зачем:** одна точка входа к данным на **один запрос**; сервисы не импортируют репозитории как синглтоны. Удобно подменять в тестах и при необходимости выполнять **несколько операций в одной транзакции** MongoDB (например удаление города и всех `WeatherRecord`).

**Минимальный контракт `UnitOfWork`:**
- Свойства или методы вроде `cities` → `CityRepository`, `weather` → `WeatherRepository` (репозитории создаются при входе в UoW или принимаются в конструкторе).
- Реализация — async context manager: `async with UnitOfWork() as uow:` → `__aenter__` / `__aexit__`; в `__aexit__` при необходимости завершить сессию транзакции.
- Опционально: метод `transaction()` — async context, внутри `await session.start_transaction()` (Motor `ClientSession`), репозитории в этой сессии вызывают Beanie с привязкой к сессии — только если реально нужна атомарность; для учебного проекта можно начать **без** транзакций и добавить на `DELETE` города.

**`dependencies.py`:**
- `async def get_uow() -> AsyncIterator[UnitOfWork]:` — `async with UnitOfWork() as uow: yield uow` (FastAPI закроет контекст после ответа).
- Роутеры не создают UoW вручную: передают `uow: UnitOfWork = Depends(get_uow)` в функцию, которая вызывает сервис `await city_service.delete(uow, city_id)`.

**Фоновая задача (`tasks.py`):** `Depends` там недоступен — в цикле явно `async with UnitOfWork() as uow:` и те же сервисы/репозитории через `uow`.

**Тесты:** `app.dependency_overrides[get_uow] = fake_uow` или фикстура, возвращающая мок UoW с подставными репозиториями.

Подробнее — раздел про UoW в [PLAN.md](PLAN.md).

**Готово, когда:** эндпоинт-заглушка с `Depends(get_uow)` открывает UoW, читает список городов через `uow.cities.get_all()` и возвращает 200.

## Шаг 7. Клиент OpenWeatherMap

- `weather_client.py` — `httpx.AsyncClient`, методы геокодинга (имя → lat/lon) и текущей погоды (lat/lon → данные под вашу модель).
- Обработка типичных ошибок API (4xx/5xx, таймаут) — хотя бы понятное исключение или код для сервиса.

**Готово, когда:** по известному городу из консоли/async-скрипта приходят координаты и погода с теми полями, что вы кладёте в `WeatherRecord`.

---

## Шаг 8. Кэш

- `cache.py` — in-memory TTL по ключу (например `city_id`), thread-safe не обязателен для одного процесса uvicorn, но структура должна быть простой для подмены позже.

**Готово, когда:** повторный запрос с тем же ключом в пределах TTL не требует обращения к OWM (проверка вручную или тестом).

---

## Шаг 9. Сервисы

- Методы принимают **`UnitOfWork`** первым аргументом (или явный `uow`), дальше — доменные параметры: `add(uow, name)`, `get_current(uow, city_id)`.
- `city_service` — геокодинг → `uow.cities.create(...)`; список; удаление через `uow.weather.delete_by_city_id` + `uow.cities.delete` (в транзакции — по желанию).
- `weather_service` — кэш → OWM → `uow.weather.save` / чтение истории через `uow.weather`.

**Готово, когда:** логика работает без роутеров (вызов из скрипта или unit-тестов с подставленными зависимостями).

## Шаг 10. Роутеры и подключение к приложению

- `routers/cities.py`, `routers/weather.py` — валидация входа, **`Depends(get_uow)`**, вызов сервисов с передачей `uow`, маппинг в схемы ответов, HTTP-коды (404, 422).
- В `main.py` — `include_router`, префикс `/api`, при необходимости CORS.

**Готово, когда:** все эндпоинты из [PLAN.md](PLAN.md) отрабатывают через Swagger `/docs`.

## Шаг 11. Фоновые задачи

- `tasks.py` — по таймеру (интервал из конфига): все города → запрос погоды → сохранение + кэш.
- Запуск и корректная отмена при shutdown — в том же месте, где вы поднимаете БД (`lifespan`, `on_event("startup")`/`("shutdown")` и т.д.): фоновая asyncio-задача с циклом и `asyncio.sleep`, при остановке приложения — `task.cancel()` и await.

**Готово, когда:** после ожидания интервала в логах/БД видны новые записи без ручных запросов.

## Частые ошибки

- Подключать роутеры до сервисов — удобнее сначала убедиться, что репозитории и клиент OWM работают изолированно.
- Хранить секреты в коде — только `.env` и переменные окружения.
- Забыть закрыть Motor-клиент при остановке приложения — явный `close()` в `shutdown` / конце `lifespan` / `atexit` (см. шаг 3).

Если что-то из шагов распухает — остановитесь на минимальной версии (например, один город вручную в БД без POST), затем добавляйте следующий слой.
